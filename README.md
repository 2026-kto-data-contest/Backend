# Backend
## Pull Request 규칙

원활한 코드 리뷰와 변경 이력 관리를 위해 아래 규칙에 따라 PR을 작성한다.

### 1. 브랜치 규칙

`main` 브랜치에서는 직접 작업하지 않고, 작업 목적에 맞는 브랜치를 생성한다.

```bash
git checkout main
git pull origin main
git checkout -b 브랜치명
```

브랜치명은 **영문 소문자와 하이픈(`-`)**을 사용한다.

| 유형 | 설명 | 예시 |
|---|---|---|
| `feat/` | 새로운 기능 추가 | `feat/user-login` |
| `fix/` | 버그 수정 | `fix/token-expiration` |
| `refactor/` | 기능 변경 없는 코드 개선 | `refactor/user-service` |
| `docs/` | 문서 수정 | `docs/update-readme` |
| `test/` | 테스트 코드 추가 및 수정 | `test/user-service` |
| `chore/` | 설정, 패키지, 빌드 관련 작업 | `chore/update-dependencies` |

---

### 2. PR 제목 규칙

PR 제목은 다음 형식으로 작성한다.

```text
[유형] 작업 내용
```

예시:

```text
[Feat] 회원가입 API 구현
[Fix] 액세스 토큰 만료 처리 오류 수정
[Refactor] 사용자 조회 로직 개선
[Docs] API 실행 방법 추가
```

| 유형 | 설명 |
|---|---|
| `Feat` | 새로운 기능 추가 |
| `Fix` | 버그 수정 |
| `Refactor` | 코드 리팩터링 |
| `Docs` | 문서 수정 |
| `Test` | 테스트 코드 추가 및 수정 |
| `Chore` | 환경 설정 및 기타 작업 |

---

### 3. PR 본문 규칙

PR 본문에는 작업 내용, 작업 이유, 테스트 내용, 참고 사항을 작성한다.

```md
## 작업 내용

- 구현하거나 수정한 내용을 작성한다.
- 주요 변경 사항을 항목별로 정리한다.

## 작업 이유

- 해당 작업이 필요한 이유를 작성한다.
- 관련 문제나 배경을 설명한다.

## 테스트 내용

- 테스트한 API 또는 기능을 작성한다.
- 테스트 결과를 간단히 작성한다.

## 참고 사항

- 리뷰어가 확인해야 할 내용을 작성한다.
- 관련 이슈가 있다면 연결한다.

Closes #이슈번호
```

작성 예시:

```md
## 작업 내용

- 이메일과 비밀번호를 이용한 로그인 API를 구현했다.
- 로그인 성공 시 액세스 토큰과 리프레시 토큰을 반환하도록 구현했다.
- 비밀번호가 일치하지 않을 경우 예외를 반환하도록 처리했다.

## 작업 이유

- 인증이 필요한 API를 사용하기 위해 로그인 기능이 필요하다.

## 테스트 내용

- 정상 로그인 요청 테스트 완료
- 존재하지 않는 이메일 요청 테스트 완료
- 비밀번호 불일치 요청 테스트 완료

## 참고 사항

- 토큰 재발급 기능은 별도 PR에서 구현할 예정이다.

Closes #12
```

### 4. PR 작성 기준

- 하나의 PR에는 하나의 작업만 포함한다.
- 기능 추가와 대규모 리팩터링을 하나의 PR에 함께 포함하지 않는다.
- 변경된 코드와 관계없는 파일은 수정하지 않는다.
- 사용하지 않는 코드, 주석, 디버깅 출력문은 제거한다.
- API가 변경된 경우 관련 문서도 함께 수정한다.
- 환경변수, 비밀번호, API 키 등의 민감한 정보는 커밋하지 않는다.
- PR 생성 전 최신 `main` 브랜치 내용을 반영한다.
- 로컬에서 애플리케이션 실행 및 테스트를 완료한 후 PR을 생성한다.

---

### 5. 커밋 메시지 규칙

커밋 메시지는 다음 형식으로 작성한다.

```text
유형: 작업 내용
```

예시:

```text
feat: 회원가입 API 구현
fix: 토큰 만료 예외 처리
refactor: 사용자 조회 로직 분리
docs: 로컬 실행 방법 추가
test: 로그인 서비스 테스트 추가
chore: JWT 라이브러리 추가
```

| 유형 | 설명 |
|---|---|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 코드 리팩터링 |
| `docs` | 문서 수정 |
| `test` | 테스트 코드 추가 및 수정 |
| `chore` | 환경 설정 및 기타 작업 |

---

### 6. 코드 리뷰 규칙

- 최소 1명 이상의 승인을 받은 후 병합한다.
- 리뷰 요청을 받은 사람은 변경 목적과 영향을 확인한다.
- 수정 요청이 있는 경우 해당 내용을 반영한 뒤 다시 리뷰를 요청한다.
- 질문이나 의견에는 답변을 남긴다.
- 리뷰가 완료되지 않은 PR은 임의로 병합하지 않는다.
- 작성자는 반영이 완료된 리뷰 코멘트를 직접 해결 처리한다.

리뷰 코멘트는 다음과 같이 구분할 수 있다.

```text
MUST: 반드시 수정이 필요한 내용
SUGGESTION: 수정하면 더 좋은 내용
QUESTION: 의도나 동작을 확인하기 위한 질문
NIT: 사소한 스타일 관련 의견
```

---

### 7. 병합 규칙

- 기본 병합 방식은 `Squash and merge`를 사용한다.
- PR 제목은 병합 후 커밋 메시지로 사용할 수 있도록 명확하게 작성한다.
- 충돌이 발생한 경우 PR 작성자가 직접 해결한다.
- 병합 후 작업 브랜치는 삭제한다.
- 테스트 또는 빌드가 실패한 상태에서는 병합하지 않는다.

---

### 8. PR 체크리스트

PR 생성 전 아래 항목을 확인한다.

- [ ] 작업 목적에 맞는 브랜치를 생성했는가?
- [ ] 하나의 PR에 하나의 작업만 포함했는가?
- [ ] 불필요한 코드와 디버깅 출력문을 제거했는가?
- [ ] 로컬에서 정상적으로 실행되는지 확인했는가?
- [ ] 테스트를 수행했는가?
- [ ] API 변경 사항을 문서에 반영했는가?
- [ ] 민감한 정보가 포함되지 않았는가?
- [ ] 관련 이슈를 연결했는가?
- [ ] 최신 `main` 브랜치를 반영했는가?
